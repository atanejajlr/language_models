An undamped free vibration analysis of a structural system should ALWAYS result in real Eigen values. There could be situations, however, wherein an undamped free vibration analysis has complex eigen values. This surely is incorrect and requires some amount of investigation / troubleshooting and insight of the problem.

This article enumerates some common reasons for complex Eigen values in an undamped free vibration analysis and the consequent troubleshotting that could be carried out to correct the anomalies.

What could be wrong if an undamped free vibration problem gives complex eigen values in a software and how to get on troubleshooting the problem?

Evaluate the determinant of the stiffness matrix

The situation normally indicates a problem with the stiffness matrix which could be due to various reasons.One thing that might be done is to check the determinant of the stiffness matrix. I have come across cases where the stiffness matrix had an infinite determinant. This physically means that the inverse stiffness matrix (i.e. the compliance or the flexibility matix) has some zero eigen values.

Examine the flexibility / compliance matrix of the system

An infinite determinant means that the inverse stiffness matrix has some zero eigen values as stated above.

If the software has capabilities to get the flexibility (compliance) matrix one can get the flexibility matrix and if the flexibility matrix has some zero eigen values then one can look at the corresponding eigen vectors. These eigen vectors physically describe the loads that are producing zero displacements. Thus, one can physically judge why the system is not displacing due to these loads (i.e. the eigen vectors corresponding to zero eigen frequency of the flexibility matrix) and correct any finite element modelling related errors in the system.

Note that there is a chance that the problem is with the mass matrix as well (since the eigen value computation depends upon both the coefficient matrices: mass and stiffness) but the siutaion is less likely as computation of amss matrix is straight forward and most often one assumes a lumped mass matrix than distributed (consistent) mass matrix.

Does your FE model have rigid elements connected?

It is also possible that if one has rigid elements in the system and the software has computed the stiffness matrix by assigning a very large value of the Young's modulus or geometric properties of the finite element has led to an infinite determinant.

This is because if some part of the calculation involves values that are very large or very small, the round-off errors in the calculation can be expanded by a huge amount and become significant. It is possible that the result of the calculation is actually mostly composed of hugely expanded round-off errors, and in that case, imaginary parts which were supposed to cancel perfectly to zero did not cancel exactly due to round-off errors in the calculation could be expanded to a large size. This would give apparent complex eigenvalues, which are really nothing but expanded round-off errors.

It is always better that rigid elements be handled as motion constraints in a FE software than movable elements. By motion constraints instead of rigid elements, the calculation model imposes constraints on the translations / rotations of the nodes than connecting an infiniteley stiff rigid element. This would have the same effect as the "physical rigid element" in the FE model but this would probably avoid problems like an "infinite determinant". Different software's have different approaches to treat the modelling of rigid elements / constraining motion.

Thus, if possible one can get on reducing the stiffness of the rigid element and see if you can get rid of the complex Eigen values because the complex Eigen values contraindicate the physics of the problem.

Is one your coefficient matrix unsymmetrical?

One should also check that both the coefficient matrices are symmetric as an unsymmetric mass or stiffness coefficient matrix will most often result in complex Eigen values.
